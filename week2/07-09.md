# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라,
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
    - `/1`, `/2` 같은 article id 기반 메인 페이지 요청에서 현재 글 기준 이전 글/다음 글 링크 동적 렌더링 구현
    - 댓글 작성 페이지 이동 시 `articleId`, `returnUrl`을 전달하고 hidden input으로 보존한 뒤, 댓글 작성 후 원래 게시글로 돌아가도록 구현
    - `JdbcTemplate -> DataSource -> Connection` 구조로 DB 연결 책임을 재정리하고, `ApplicationContext`와 `RequestMapper`로 객체 생성/조합 책임과 요청 매핑 책임을 분리

- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
    - `returnUrl` 검증은 내부 경로만 허용하도록 최소 방어를 넣었지만, 더 복잡한 query string 보존이나 허용 경로 정책은 아직 정교하게 분리하지 못했다.
    - `ApplicationContext`가 객체 조립 책임을 갖게 되었지만, 테스트용 context와 운영용 context를 나누는 수준까지는 확장하지 않았다.
    - 복잡한 서비스 로직의 트랜잭션 처리를 구현하지 못하였다.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
    - `HttpRequestResolver`가 DAO, Service, Cache까지 직접 생성하는 구조를 없애고, 객체 생성과 조합은 `ApplicationContext`, 요청 매핑은 `RequestMapper`, 실제 요청 처리는 `RequestHandler`가 담당하도록 분리했다.

- 그렇게 판단한 이유:
    - 기존 `HttpRequestResolver`는 이름은 resolver였지만 실제로는 컨테이너/레지스트리처럼 동작하고 있었다.
    - 요청 path에 맞는 service를 찾는 책임과, DAO/Service/Cache/DataSource를 생성하고 연결하는 책임이 한 클래스에 섞여 있어 변경 이유가 많았다.
    - 특히 `JdbcTemplate`을 `DataSource` 기반으로 바꾸면서 DB 연결 객체를 여러 DAO가 공유해야 했기 때문에, 조합 책임을 별도 객체로 빼는 것이 더 자연스럽다고 판단했다.

- 고려했지만 선택하지 않은 대안:
    - `JdbcTemplate`에 계속 `String jdbcUrl`만 넘기는 방식
        - 구현은 단순하지만 `JdbcTemplate`이 연결 생성 방식까지 알아야 해서 책임이 넓어진다.

- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
    - `JdbcTemplate`은 `DataSource`에서 `Connection`을 얻는 역할에 집중하게 되었고, 연결 생성 방식은 `H2DataSourceFactory`로 이동했다.
    - DAO 테스트들이 URL 문자열이 아니라 같은 `DataSource`를 공유하도록 바뀌면서 실제 운영 구조와 테스트 구조가 더 가까워졌다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 여러 DB 작업을 하나의 기능 안에서 처리할 때, 일부 작업만 성공하고 일부 작업은 실패할 수 있다는 문제를 확인했다. 예를 들어 댓글 저장은 성공했지만 댓글 수 증가는 실패하면서 정합성이 깨질 수 있음을 확인했고 트랜잭션을 관리해야 함을 느꼈다.
- 처음 추측한 원인: 각각의 DAO가 독립적으로 SQL을 실행하면 된다고 생각했지만, 여러 SQL이 하나의 작업 단위로 묶여야 하는 경우에는 단순 실행만으로는 부족하다고 판단했다.
- 원인을 확인하기 위해 시도한 방법: `autoCommit`, `commit`, `rollback`의 동작을 확인하고, 같은 작업 단위에 포함되는 SQL들이 같은 `Connection`을 사용해야 한다는 점을 정리했다. 또한 트랜잭션 시작과 종료 책임을 어디에 둘지 고민했다.
- 실제 원인: 트랜잭션은 단순히 SQL을 실행하는 기능이 아니라, 여러 DB 작업을 하나의 논리적 작업 단위로 묶는 장치였다. 기존 방식처럼 각 DAO가 연결과 실행을 따로 관리하면 하나의 기능을 원자적으로 처리하기 어려웠다.
- 해결 방법: 트랜잭션 경계를 서비스 흐름에 두고, 연결 생성, `autoCommit(false)`, `commit`, `rollback`, 자원 해제는 별도의 트랜잭션 관리 객체가 담당하도록 분리하는 방향을 잡았다. DAO는 SQL 실행에 집중하고, 트랜잭션 처리는 별도의 관심사로 분리해야 한다는 점을 이해했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 하나의 기능이 여러 DB 작업을 포함하는지, 그 작업들이 함께 성공하거나 함께 실패해야 하는지 먼저 확인하겠다. 그리고 같은 작업 단위 안의 SQL들이 같은 연결을 공유하고 있는지, 트랜잭션 경계가 명확한지 먼저 점검하겠다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 트랜잭션이 왜 필요한지와 어떤 원리로 동작하는지까지 파고들었다. 특히 여러 SQL을 하나의 작업 단위로 묶기 위해 같은 커넥션을 유지해야 한다는 점과, 현재 실행 흐름에서 커넥션을 공유하기 위해 스레드 로컬을 사용할 수 있다는 점을 학습했다. 또한 리플렉션, 데코레이터 패턴, 프록시 패턴을 함께 학습하면서 자바 생태계에서 공통 관심사를 어떻게 분리하고 적용할 수 있는지 이해했다.
- 그 접근이 도움이 된 이유: 트랜잭션이 데이터 정합성을 지키기 위한 작업 단위라는 점을 이해했다. 여러 SQL을 같은 연결로 실행하고, 모두 성공하면 커밋하고 하나라도 실패하면 롤백해야 한다는 흐름을 구체적으로 이해할 수 있었다. 또한 프록시를 사용하면 핵심 비즈니스 로직을 수정하지 않고도 트랜잭션 처리, 로깅, 권한 확인 같은 부가 기능을 외부에서 감쌀 수 있다는 점을 알게 되었다.
- 다음에도 반복하고 싶은 점: 기능을 구현할 때 단순히 동작 여부만 확인하지 않고, 실패했을 때 데이터가 어떤 상태로 남는지까지 함께 확인하겠다. 또한 코드가 지저분해지는 지점이 생기면 단순히 줄이는 데 집중하기보다, 어떤 관심사가 섞였는지 먼저 확인하고 분리할 수 있는 구조를 고민하겠다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 처음에는 트랜잭션을 적용하면 코드가 지저분해지는 문제에만 집중했다. 그보다 먼저 트랜잭션이 왜 필요한지, 어떤 작업들을 하나로 묶어야 하는지부터 정리했어야 했다.
- 그렇게 된 원인: 트랜잭션을 구조 설계의 문제라기보다 단순히 `commit`, `rollback`을 어디에 작성할지의 문제로 좁게 바라봤다. 그래서 비즈니스 로직, SQL 실행, 트랜잭션 관리 책임이 섞일 수 있다는 점을 늦게 인식했다.
- 다음에는 다르게 해볼 구체적인 행동: 먼저 가장 단순한 방식으로 트랜잭션 흐름을 구현해보고, 이후 반복되는 연결 관리와 예외 처리 코드를 별도 객체로 분리하겠다. 앞으로는 비즈니스 로직은 서비스가 담당하고, SQL 실행은 DAO가 담당하며, 트랜잭션 시작과 종료는 별도의 관리 객체가 담당하도록 관심사를 나누겠다.
