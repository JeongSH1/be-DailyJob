# be-DailyJob 

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: HTTP Request와 HTTP Response를 효율적으로 관리하고 확장 가능하게 만드는 구조를 고민하고 구현했다. `HttpRequest`는 메서드, 경로, 프로토콜, 헤더, 쿼리 파라미터, 본문을 관리하도록 구성했고, `HttpResponse`는 상태 코드, 헤더, 본문을 조합해 응답을 생성하도록 구현했다. 또한 `200 OK`, `201 Created`, `404 Not Found`, `500 Internal Server Error`, `302 Redirect` 같은 응답을 메서드로 표현할 수 있도록 정리했다.

- (오늘의 목표 중에서) 아직 끝내지 못한 부분: HTTP 외의 다른 프로토콜까지 확장 가능한 구조는 구현하지 못했다. 또한 Request와 Response의 공통 추상화를 어디까지 가져갈지에 대한 결론은 냈지만, 모든 구조를 완전히 정리하지는 못했다.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 요청과 응답을 단순 문자열로 직접 처리하지 않고, `HttpRequest`와 `HttpResponse` 객체로 분리해 관리하기로 했다.

- 그렇게 판단한 이유: 요구사항이 계속 쌓이는 구조이기 때문에, 요청이 들어올 때마다 문자열을 직접 파싱하고 응답 문자열을 직접 조립하면 변경 지점이 많아질 것이라고 판단했다. 요청은 메서드, 경로, 헤더, 파라미터처럼 의미 있는 단위로 관리하고, 응답은 상태 코드, 헤더, 본문을 분리해 관리하는 편이 유지보수에 더 적합하다고 보았다.

- 고려했지만 선택하지 않은 대안: `RequestHandler`에서 요청 문자열을 직접 파싱하고, 성공이나 실패 응답 문자열을 바로 출력하는 방식도 고려했다. 이 방식은 당장 요구사항을 빠르게 구현하기에는 좋지만, 이후 정적 리소스, 리다이렉트, 에러 응답이 추가될 때 코드가 복잡해질 수 있어 선택하지 않았다.

- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 요청과 응답의 책임이 분리되면서 코드 구조가 더 명확해졌다. `HttpResponse`에서 상태 라인, 헤더, 빈 줄, 본문 순서로 응답을 조립하게 되어 HTTP 응답 형식을 안정적으로 관리할 수 있었다. 다만 확장성을 크게 고려하면서 실제 요구사항 구현 속도는 느려졌다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: HTTP 요청과 응답을 확장 가능한 구조로 만들고 싶다는 생각이 커지면서, 실제 요구사항보다 더 넓은 범위까지 설계하려고 했다. HTTP뿐 아니라 다른 프로토콜의 `Request`, `Response`가 들어와도 처리 가능한 구조를 고민했고, 그 과정에서 실제 구현이 느려졌다.

- 처음 추측한 원인: 처음에는 좋은 객체지향 구조를 만들기 위해 필요한 고민이라고 생각했다. `Request`와 `HttpRequest`, `Response`와 `HttpResponse`를 어떻게 추상화할지, record를 써도 되는지, 인터페이스나 상속을 어떻게 적용할지 계속 고민했다.

- 원인을 확인하기 위해 시도한 방법: 현재 요구사항을 다시 확인했다. 오늘 필요한 것은 HTTP 기반의 회원가입 요청을 처리하고, 정적 리소스와 기본 HTTP 응답을 안정적으로 반환하는 것이었다. 그런데 나는 아직 요구되지 않은 다른 프로토콜이나 다양한 Request/Response 타입까지 고려하고 있었다.

- 실제 원인: 현재 요구사항의 변화 가능성보다 더 큰 수준의 확장성을 미리 설계하려고 한 것이 원인이었다. 유지보수성을 높이려는 의도는 좋았지만, 실제로 필요한 추상화와 아직 필요하지 않은 추상화를 구분하지 못했다.

- 해결 방법: 우선 범위를 HTTP 요청과 HTTP 응답으로 좁혔다. `Request`, `Response` 전체를 일반화하기보다, 현재 필요한 `HttpRequest`, `HttpResponse`를 먼저 안정적으로 구현하는 데 집중했다. 응답도 상태 코드, 헤더, 본문을 조합할 수 있는 정도로 확장성을 확보하고, 다른 프로토콜까지 고려한 구조는 뒤로 미뤘다.

- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 새로운 추상화를 만들기 전에 “현재 요구사항에서 실제로 변하는 부분인가?”를 먼저 확인하겠다. 아직 등장하지 않은 변화까지 미리 일반화하기보다, 같은 형태의 중복이나 변경 지점이 실제로 드러났을 때 추상화를 적용하겠다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: HTTP 요청과 응답을 단순 문자열 처리로만 보지 않고, 각각의 책임을 객체로 나누어 관리하려고 했다.

- 그 접근이 도움이 된 이유: HTTP 응답은 상태 라인, 헤더, 빈 줄, 본문이라는 명확한 형식이 있기 때문에 이를 `HttpResponse` 안에서 관리하니 코드가 더 읽기 쉬워졌다. 요청도 메서드, 경로, 헤더, 파라미터로 분리하니 서비스 코드가 HTTP 문자열 자체를 몰라도 되었다.

- 다음에도 반복하고 싶은 점: 요구사항이 추가될 가능성이 있는 부분은 단순 문자열 처리에 머무르지 않고, 의미 있는 객체로 분리하는 습관을 유지하겠다. 다만 분리의 범위는 현재 요구사항을 기준으로 조절하겠다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 너무 완벽한 설계에 매몰되었다. 현재는 HTTP 요청과 응답만 처리하면 되는 상황인데, HTTP가 아닌 다른 프로토콜이 들어왔을 때도 처리 가능한 구조, 다른 `Request`, `Response` 타입도 확장 가능한 구조까지 고민했다. 그러다보니 정작 기능 구현하는데 깊은 고민을 하지 못했다.

- 그렇게 된 원인: 기존 코드를 변경하지 않고 손쉽게 확장하고 싶다는 생각이 강했다. 요구사항이 계속 쌓이는 상황에서 처음부터 좋은 구조를 만들고 싶었지만, 그 과정에서 현재 필요한 구현 범위를 넘어서는 추상화까지 고민하게 되었다.

- 다음에는 다르게 해볼 구체적인 행동: 먼저 현재 요구사항을 만족하는 가장 단순한 구조로 구현하고, 중복이나 변경 가능성이 실제로 드러난 지점만 리팩터링하겠다. 추상화를 만들기 전에는 “이 변화가 지금 실제로 필요한가?”, “테스트나 기능 구현을 늦출 만큼 가치가 있는가?”를 확인하겠다.
