# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - `multipart/form-data`가 텍스트와 파일처럼 서로 다른 종류의 데이터를 하나의 HTTP 요청으로 전달하기 위해 필요한 형식이라는 점을 학습했다.
  - boundary와 각 part의 헤더, 본문 구조를 이해하고 이를 직접 파싱하는 코드를 구현했다.
  - HTTP 헤더는 문자열로 파싱하되, 파일이 포함될 수 있는 body는 `byte[]`로 유지하도록 요청 파서 구조를 수정했다.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - 비정상적인 multipart 요청과 여러 파일이 같은 이름으로 전달되는 경우에 대한 예외 처리와 테스트가 남아 있다.

### 2. 오늘 가장 의미있었던 기술적 판단
multipart parser만 별도로 추가하는 것이 아니라, HTTP 요청을 읽는 구조부터 수정해야 한다고 판단했다. 기존에는 전체 요청을 문자열로 변환했지만, multipart body에는 이미지나 파일과 같은 바이너리 데이터가 포함될 수 있어 문자열 변환 과정에서 원본 바이트가 손상될 수 있었다.

따라서 요청 라인과 헤더 영역만 문자열로 변환하고, body는 `Content-Length`만큼 `byte[]`로 읽어 유지하도록 변경했다. 이 구조를 선택하면서 요청을 읽는 역할과 Content-Type에 따라 body를 해석하는 역할도 자연스럽게 분리할 수 있었다.

앞으로도 세부 기능을 구현하기 전에 현재 데이터 표현 방식이 기능의 요구사항에 적합한지 먼저 확인하겠다. 특히 파일, 네트워크, 인코딩처럼 데이터 손실 가능성이 있는 기능은 타입과 변환 시점을 우선 검토하겠다.

### 3. 가장 막혔던 문제와 해결 과정
처음에는 boundary를 기준으로 body 문자열을 나누면 multipart를 처리할 수 있다고 생각했다. 그러나 parser를 구현하는 과정에서 body를 문자열로 바꾼 뒤 다시 바이트 배열로 복원하면 파일 데이터가 손상될 수 있다는 문제를 확인했다.

문제의 원인은 boundary 탐색 로직이 아니라, HTTP 요청 전체를 UTF-8 문자열로 읽는 기존 구조에 있었다. 이를 해결하기 위해 헤더 종료 지점인 `\r\n\r\n`까지만 문자열로 변환하고, 이후 body는 원본 바이트 그대로 읽도록 구조를 변경했다. 그 후 boundary 추출, part 분리, part 헤더 파싱, 파일 저장 순서로 기능을 나누어 구현했다.

다음에 비슷한 문제가 생기면 구현 코드부터 작성하기보다 먼저 입력 데이터가 텍스트인지 바이너리인지, 인코딩 변환이 안전한지, 데이터의 시작과 끝을 어떤 기준으로 판단해야 하는지를 확인하겠다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
오늘은 multipart parser 코드를 단순히 완성하는 데 그치지 않고, `multipart/form-data`가 왜 존재하고 어떤 문제를 해결하기 위해 필요한지부터 학습했다.

기존의 `application/x-www-form-urlencoded`는 텍스트 기반 데이터 전달에는 적합하지만 파일과 같은 바이너리 데이터를 함께 보내기에는 불편하다. JSON에 Base64로 파일을 포함하는 방법도 있지만 데이터 크기가 증가하고 별도의 인코딩과 디코딩이 필요하다. multipart는 각 데이터를 part 단위로 분리하고 part마다 이름, 파일명, Content-Type을 표현할 수 있어 폼 데이터와 파일을 함께 전송하기에 적합하다는 점을 이해했다.

이 배경을 먼저 학습했기 때문에 boundary가 필요한 이유, 각 part가 자체 헤더를 가지는 이유, body를 `byte[]`로 다뤄야 하는 이유를 연결해서 이해할 수 있었다. 앞으로도 새로운 기술을 사용할 때 사용법보다 먼저 등장 배경, 해결하려는 문제, 대안과 장단점을 확인하겠다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
multipart parser의 바이트 탐색과 인덱스 계산이 복잡하게 느껴져 AI에게 구현 코드의 도움을 받았다. 구현 속도는 빨라졌지만, 처음에는 AI가 제시한 코드를 이해하기보다 현재 코드에 적용하는 데 집중했다.

이후 실제 multipart 요청 예시를 기준으로 boundary, CRLF, part header, part body의 위치를 직접 확인하면서 코드의 의미를 다시 이해했다. 앞으로는 AI에게 전체 구현을 바로 요청하기보다 내가 먼저 파싱 순서를 정리하고, 해결하지 못한 부분이나 잘못 이해한 부분만 질문하겠다.

또한 AI가 작성한 코드는 그대로 사용하지 않고, 각 메서드가 왜 필요한지와 인덱스가 어떤 기준으로 이동하는지를 설명할 수 있을 때 적용하겠다.

# 주간 프롬프트 회고

### 1. 내가 최종 선택한 가장 잘 작성했다고 생각하는 프롬프트
원본 프롬프트:

> multipart/form-data에 대해 공부하고 싶다.  
> 왜 존재하는지, 왜 필요한지, 대안이 있는지, 어떤 이점이 있는지, 어떤 형식인지 설명해주고 parser 구현 방법도 알려줘.

선택한 이유:

단순히 parser 코드를 요청하지 않고, 기술의 등장 이유와 필요성, 대안, 형식까지 함께 질문했다. 이번 학습의 목적은 multipart 기능을 빠르게 붙이는 것이 아니라 직접 parser를 구현하면서 HTTP body가 실제로 어떤 구조로 전달되는지 이해하는 것이었기 때문에 적절한 프롬프트라고 판단했다.

### 2. 원본 프롬프트 구조 분석
원본 프롬프트에는 multipart의 개념과 구현 방법을 함께 알고 싶다는 의도는 담겨 있었다. 하지만 순수 Java 웹 서버를 만들고 있다는 상황, 기존에는 전체 요청을 문자열로 처리하고 있었다는 구조, 외부 라이브러리 없이 직접 구현해야 한다는 제약은 포함되어 있지 않았다.

또한 완성 코드보다 파싱 원리와 각 데이터의 경계를 이해하고 싶다는 학습 목적이 충분히 드러나지 않았다. 이 때문에 AI가 설명보다 전체 구현 코드를 먼저 제공할 가능성이 있었다.

### 3. AI에게 프롬프트를 개선 요청하기
- 프롬프트 개선안

```text
순수 Java로 구현한 미니 웹 서버에서 파일 업로드를 지원하기 위해
multipart/form-data parser를 직접 구현하려고 해.

현재 HttpRequestParser는 InputStream에서 HTTP 요청 전체를 읽어
UTF-8 String으로 변환한 뒤 요청 라인, 헤더, body를 분리하고 있어.

다음 내용을 중심으로 설명해.

1. multipart/form-data가 왜 필요하며 어떤 문제를 해결하는지
2. application/x-www-form-urlencoded, JSON, Base64 전송과 비교했을 때의 장단점
3. boundary, Content-Disposition, Content-Type, CRLF를 포함한 실제 요청 형식
4. 파일이 포함된 body를 String이 아닌 byte[]로 처리해야 하는 이유
5. InputStream에서 헤더와 body를 분리하고 multipart part를 파싱하는 순서
6. boundary 누락이나 잘못된 part 형식에 필요한 예외 처리와 테스트 방법

외부 multipart 라이브러리는 사용하지 않고 Java 표준 라이브러리만 사용해.
완성 코드를 바로 제시하기보다 각 파싱 단계가 왜 필요한지와 바이트 인덱스가 어떻게 이동하는지 먼저 설명해주세요.
