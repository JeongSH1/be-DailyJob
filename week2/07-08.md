# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - 동적 HTML 응답 흐름을 구현했다.
  - 인증/인가 처리를 위한 공통 서비스 흐름을 `HttpService` 추상 클래스로 정의했다.
  - `process()`에서 인증 확인 -> 인증 실패 처리 -> 실제 서비스 실행 순서를 고정했다.
  - 세부 동작은 `checkAuth()`, `doNotSatisfiedAuthCondition()`, `doService()`에서 구현하도록 분리했다.
  - 세션 쿠키의 `sid`를 기준으로 로그인 여부를 확인하는 `isLogin()` 공통 로직을 만들었다.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - 게시글 CRUD 구현을 완료하지 못했다.
  - 서비스별 인증 조건 테스트를 충분히 작성하지 못했다.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - 인증/인가 과정에서 반복되는 실행 흐름을 추상 클래스와 템플릿 메서드 패턴으로 분리했다.
  - 개별 서비스가 직접 인증 조건문을 반복하지 않고, 부모 클래스의 `process()` 흐름을 따르도록 했다.
- 그렇게 판단한 이유:
  - 각 서비스의 `doService()` 내부에서 인증을 직접 확인하면 중복이 생기고, 인증 검사를 빠뜨릴 수 있다고 판단했다.
  - 요청 처리 순서는 공통이지만, 인증 필요 여부와 실패 응답은 서비스마다 달라질 수 있어 `checkAuth()`와 `doNotSatisfiedAuthCondition()`으로 분리했다.
- 고려했지만 선택하지 않은 대안:
  - 서비스마다 직접 로그인 여부를 검사하는 방식은 단순하지만 중복이 커질 수 있어 선택하지 않았다.
  - 인증 필터나 인터셉터를 별도 객체로 두는 방식은 더 깔끔할 수 있지만, 현재 단계에서는 직접 흐름을 체감하는 것이 우선이라고 판단했다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - `process()`만 보면 인증 확인 -> 실패 처리 -> 실제 실행 흐름을 한눈에 볼 수 있게 되었다.
  - 다만 부모 클래스 변경이 모든 하위 서비스에 영향을 줄 수 있어, 상속 구조의 유지보수 부담도 함께 체감했다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - 인증이 필요한 서비스와 필요하지 않은 서비스가 섞이면서 인증 검사를 어디에 둘지 고민했다.
  - 각 서비스에 인증 조건문을 직접 넣으면 비슷한 코드가 반복될 것 같았다.
- 처음 추측한 원인:
  - 서비스별 기능 구현과 인증/인가 흐름이 섞여 있어 중복이 생긴다고 생각했다.
- 원인을 확인하기 위해 시도한 방법:
  - 요청 처리 과정을 인증 확인, 인증 실패 처리, 실제 서비스 실행 단계로 나누어 보았다.
  - 공통으로 고정할 흐름과 서비스마다 달라질 동작을 구분했다.
- 실제 원인:
  - 문제의 핵심은 로그인 확인 로직 자체보다, 인증을 언제 검사하고 실패 시 어디서 중단할지에 대한 공통 흐름이었다.
- 해결 방법:
  - `process()`에 공통 실행 순서를 두고, 인증 판단은 `checkAuth()`, 실패 응답은 `doNotSatisfiedAuthCondition()`, 실제 기능은 `doService()`로 분리했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 반복되는 것이 단순 코드인지, 반드시 지켜야 하는 실행 흐름인지 먼저 구분한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - 인증 조건문을 각 서비스에 복사하지 않고, 공통 실행 흐름으로 분리했다.
  - 템플릿 메서드 패턴을 실제 코드에 적용해보았다.
- 그 접근이 도움이 된 이유:
  - 인증 검사를 빼먹을 가능성이 줄었고, 인증 실패 시 실제 서비스 로직으로 넘어가지 않도록 흐름을 통제할 수 있었다.
  - 공통 로그인 확인 로직을 `isLogin()`에 모아 수정 범위를 줄일 수 있게 되었다.
- 다음에도 반복하고 싶은 점:
  - 중복 코드가 보이면 바로 제거하기보다, 공통 흐름인지 공통 유틸인지 먼저 구분하겠다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
  - 상속의 장점은 체감했지만, 상속이 만드는 강한 결합은 처음부터 충분히 고려하지 못했다.
  - `checkAuth()`, `doNotSatisfiedAuthCondition()`, `doService()`처럼 하위 클래스가 따라야 할 규칙이 늘어났다.
- 그렇게 된 원인:
  - 당장 반복 코드를 줄이고 요청 처리 흐름을 통일하는 데 집중했고, 상속을 “중복 제거 도구”로 먼저 바라봤고, “공통 흐름을 반드시 강제해야 하는가”라는 질문을 늦게 던졌다.
- 다음에는 다르게 해볼 구체적인 행동:
  - 상속을 사용하기 전에 공통 코드가 아니라 공통 실행 흐름이 필요한지 먼저 확인하겠다.
  - 인증/인가처럼 여러 서비스에 걸쳐 적용되는 관심사는 추후 별도 인증 객체나 필터 구조로 분리할 수 있는지 검토하겠다.
