# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라,  
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.  
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - 기존 `HttpService` 성격의 요청 처리 객체를 `Servlet` 구조로 변경하고, HTTP 메서드에 따라 `doGet`, `doPost`, `doPut`, `doDelete`가 실행되도록 정리했다.
  - 요청 경로에 맞는 서블릿을 선택하도록 `RequestMapper`를 `ServletMapper`로 변경했다.
  - `ApplicationContext`를 `ServletContext`로 변경하고, 서블릿 생성과 의존성 연결, 서블릿 목록 관리를 한곳으로 모았다.
  - 스프링 컨테이너, 서블릿 컨테이너, WAS, 톰캣의 역할 차이를 학습했다.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - 현재 구조가 실제 서블릿 표준을 어디까지 따라야 하는지 기준을 명확히 세우지 못했다.
  - 페이지 처리 로직과 비즈니스 로직의 책임은 아직 완전히 분리하지 못했다.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 현재 구현 대상을 스프링 MVC가 아니라 요청 수신, 매핑, 실행을 담당하는 단순한 WAS 구조로 정의하고 객체 이름과 책임을 이에 맞게 변경했다.
- 그렇게 판단한 이유: `Service`, `RequestMapper`, `ApplicationContext`라는 이름을 사용하자 빈 관리와 DI, 핸들러 어댑터 같은 스프링 구조를 구현하는 방향으로 사고가 흘렀다. 하지만 현재 필요한 것은 요청 경로에 맞는 처리 객체를 찾아 실행하는 서버 구조였다.
- 고려했지만 선택하지 않은 대안: 기존 이름을 유지한 채 내부 동작만 변경하는 방법도 고려했지만, 이름과 실제 책임이 다르면 이후 설계 판단에서도 계속 혼란이 발생할 수 있어 선택하지 않았다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 요청 수신은 `WebServer`, 요청 분배는 `ServletMapper`, 실행은 `Servlet`, 구성은 `ServletContext`가 담당하도록 경계가 명확해졌다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 구현을 진행할수록 현재 만드는 것이 스프링 컨테이너인지, 톰캣과 같은 서블릿 컨테이너인지 구분하기 어려워졌다.
- 처음 추측한 원인: 객체 이름만 바꾸면 구조가 자연스럽게 정리될 것이라고 생각했다.
- 원인을 확인하기 위해 시도한 방법: 톰캣, WAS, 서블릿 컨테이너, `ApplicationContext`, `ServletContext`가 각각 무엇을 관리하는지 요청 흐름을 기준으로 비교했다.
- 실제 원인: 스프링에서 익숙하게 본 이름을 그대로 사용하면서, 객체가 실제로 담당하는 책임보다 프레임워크 구조를 먼저 따라가고 있었다.
- 해결 방법: 서버가 처리해야 하는 최소 흐름을 먼저 정의하고, 그 역할에 맞춰 객체를 `Servlet`, `ServletMapper`, `ServletContext`로 변경했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 구현체의 이름을 정하기 전에 현재 만들려는 시스템의 범위와 각 객체가 관리해야 하는 대상을 먼저 적어보겠다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 단순히 클래스 이름만 변경하지 않고 톰캣, 서블릿 컨테이너, 스프링 컨테이너의 책임을 비교하며 전체 요청 흐름을 다시 정리했다.
- 그 접근이 도움이 된 이유: 톰캣은 HTTP 요청과 서블릿 실행을 관리하고, 스프링 컨테이너는 빈의 생성과 의존성 주입을 관리한다는 차이를 현재 코드에 직접 대입할 수 있었다.
- 다음에도 반복하고 싶은 점: 익숙한 프레임워크 구조를 그대로 복사하기보다, 현재 요구사항에 필요한 최소 책임을 먼저 정의하고 이름과 구조를 결정하겠다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 실제 서블릿 표준을 학습한 뒤 이를 필요 이상으로 그대로 따라가려 했다.
- 그렇게 된 원인: 톰캣의 구조를 이해한 직후라서 `Servlet`, `ServletContext`, 생명주기 같은 개념을 모두 구현해야 더 올바른 구조가 된다고 생각했다.
- 다음에는 다르게 해볼 구체적인 행동: 표준의 구조는 참고하되 현재 과제에 필요한 기능만 선택하겠다. 요청 매핑과 메서드 분기에는 서블릿 개념을 사용하되, 생명주기나 필터 등은 실제 요구가 생길 때 추가하겠다.

### 6. 전체 회고
오늘은 내가 만들고 있는 서버의 정체를 먼저 정의하는 일이 구조 설계보다 중요하다는 것을 배웠다. 처음에는 스프링에서 익숙한 이름을 사용하면서 자연스럽게 스프링 MVC와 비슷한 구조를 만들고 있었지만, 실제 목표는 요청을 받고 적절한 처리 객체를 실행하는 단순한 WAS에 가까웠다.

다만 서블릿 구조가 현재 문제를 잘 설명해준다는 이유로 실제 표준을 지나치게 따라가려 한 점은 아쉬웠다. 표준을 그대로 복제하는 것이 목표가 아니라, 표준이 왜 그런 책임을 나눴는지 이해하고 현재 요구사항에 필요한 부분만 적용해야 한다.

앞으로는 먼저 구현 범위를 한 문장으로 정의하고, 각 객체가 무엇을 관리하고 어떤 이유로 변경되는지 확인한 뒤 구조를 선택하겠다.
