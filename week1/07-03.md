# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라,
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 회원가입 요청의 쿼리 파라미터를 파싱해 사용자를 생성하고 저장하는 로직을 구현
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 요청에 적합한 Service를 찾아주는 RequestMapper가 경로 문자열과 조건문에 의존하고 있다. 새로운 요청이 추가될 때마다 기존 코드를 수정하지 않도록 라우팅 구조를 개선하고자 한다.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: RequestHandler에서 회원가입과 정적 리소스 처리 로직을 직접 분기하지 않고, 공통 Service 인터페이스와 ServiceTemplate을 통해 실행하도록 설계를 변경했다.
- 그렇게 판단한 이유: 기능이 추가될 때마다 RequestHandler를 수정하면 네트워크 처리, 요청 분석, 비즈니스 로직이 한 클래스에 섞이게 된다. 요청 처리 흐름과 개별 기능의 구현을 분리
하면 회원가입 외의 기능도 같은 실행 흐름에 추가할 수 있다고 판단했다.
- 고려했지만 선택하지 않은 대안: RequestHandler 내부에서 요청 경로를 if-else로 구분하고 회원가입 로직을 직접 호출하는 방법을 고려했다. 구현은 단순하지만 요청 종류가 늘어날수록 조건문과 의존성이 함께 증가하므로 선택하지 않았다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 회원가입 로직을 UserCreateService로 분리해 소켓 없이 단위 테스트할 수 있게 되었다. 공통 예외 처리도 ServiceTemplate에 모을 수 있었다. 다만 서비스를 선택하는 RequestMapper가 새로운 변경 지점이 되었기 때문에 라우팅 구조에 대한 추가 설계가 필요하다.

3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 쿼리스트링이 없는 /index.html 요청을 파싱할 때 ArrayIndexOutOfBoundsException이 발생해 테스트와 서버 요청 처리가 실패했다.
- 처음 추측한 원인: 처음에는 정확한 원인을 바로 파악하지 못했다. 특히 요청 처리 흐름을 try-catch로 감싸면서 예외가 콘솔에 명확히 출력되지 않아, 어떤 지점에서 문제가 발생했는지 확인하기 어려웠다.
- 원인을 확인하기 위해 시도한 방법: 먼저 예외를 숨기고 있던 try-catch 구조를 점검하고, 에러가 발생한 위치를 확인할 수 있도록 예외 메시지를 드러냈다. 이후 요청 경로를 파싱하는 로직을 단계별로 확인하면서, 쿼리스트링이 있는 요청과 없는 요청에서 배열이 어떻게 생성되는지 비교했다. 이 과정에서 AI의 도움을 받아 가능성 있는 원인을 함께 검토했다.
- 실제 원인: 요청 경로를 ?로 분리한 뒤, 모든 요청에 쿼리스트링이 존재한다고 가정하고 pathElem[1]에 접근하고 있었다. 그러나 /index.html처럼 쿼리스트링이 없는 요청에서는 분리된 배열의 길이가 1이기 때문에 pathElem[1] 접근 시 ArrayIndexOutOfBoundsException이 발생했다.
- 해결 방법: 분리된 배열의 길이를 먼저 확인하고, 쿼리스트링이 존재하며 비어 있지 않은 경우에만 파라미터를 파싱하도록 변경했다. 수정 후 쿼리스트링이 있는 요청과 없는 요청 모두에서 전체 테스트가 통과하는 것을 확인했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 입력 형식이 항상 동일하다고 가정한 코드가 있는지 먼저 확인하겠다. 특히 요청 파싱 기능처럼 다양한 입력이 들어올 수 있는 로직은 쿼리스트링의 유무, 빈 값, 잘못된 형식 등을 테스트 케이스로 먼저 작성하고 구현하겠다. 또한 예외를 무조건 try-catch로 감추기보다, 원인을 파악할 수 있도록 로그나 예외 메시지를 남기는 방식으로 디버깅하겠다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 회원가입 기능을 단순히 동작하게 만드는 데 그치지 않고, 이후 다른 요청 처리 기능이 추가될 상황을 고려해 공통 실행 흐름과 개별 서비스의 책임을 분리했다.
- 그 접근이 도움이 된 이유: RequestHandler는 소켓 입출력과 요청 처리의 전체 흐름 제어에 집중하고, 회원가입과 정적 리소스 응답은 각각의 서비스가 담당하도록 구조를 나누었다. 덕분에 특정 기능에서 문제가 발생했을 때 확인해야 할 코드 범위가 줄었고, 기능별 책임도 더 명확해졌다.
- 다음에도 반복하고 싶은 점: 기능을 추가하기 전에 기존 객체가 너무 많은 책임을 가지게 되지는 않는지 먼저 확인하겠다. 또한 분리한 객체가 독립적으로 테스트 가능한 구조인지 함께 검토하며 기능을 확장하겠다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 요청 처리 중 발생할 수 있는 예외를 깊게 고민하지 않고 try-catch로 감싸기만 했다. 그 결과 실제로 어떤 예외가 발생했는지 콘솔에서 바로 확인하기 어려웠고, /index.html 요청 파싱 중 발생한 ArrayIndexOutOfBoundsException의 원인을 파악하는 데 시간이 더 걸렸다.
- 그렇게 된 원인: 예외를 처리한다는 목적보다 서버가 중간에 종료되지 않게 막는 데만 집중했다. 어떤 예외를 어디에서 잡아야 하는지, 잡은 뒤 로그를 남기거나 적절한 응답으로 변환해야 하는지까지 설계하지 못했다. 결국 정상 흐름을 빠르게 연결하는 데 매몰되어, 예외 처리의 의미와 디버깅 가능성을 충분히 고려하지 못했다.
- 다음에는 다르게 해볼 구체적인 행동: 예외를 무조건 try-catch로 감싸기보다, 해당 위치에서 실제로 처리할 수 있는 예외인지 먼저 판단하겠다. 처리할 수 없는 예외는 상위로 전달하고, 처리해야 하는 예외는 원인을 확인할 수 있도록 로그나 메시지를 남기겠다. 또한 요청 파싱처럼 입력 형식에 따라 실패할 수 있는 로직은 쿼리스트링 없음, 빈 값, 잘못된 형식 같은 실패 케이스를 테스트로 먼저 작성하겠다.
