# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라,
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
- HTTP POST 요청의 Body를 `Content-Length` 기준으로 읽고, `application/x-www-form-urlencoded` 데이터를 파싱해 회원가입/로그인에 사용하도록 구현했다.
- 회원가입 성공 후 `302 Found + Location` 헤더를 이용해 `/index.html` 또는 `/login`으로 이동하는 redirect 흐름을 구현했다.
- 로그인 성공 시 `sid` 쿠키를 발급하고, 서버 메모리에 `sid -> username` 세션을 저장하도록 구현했다.
- 메인 페이지 요청 시 `Cookie` 헤더에서 `sid`를 읽고, 세션이 존재하면 서버에서 HTML을 동적으로 수정해 `안녕하세요! {username}님!` 문구와 `글쓰기/로그아웃` 메뉴를 내려주도록 구현했다.

- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
- `sid` 쿠키 값은 랜덤 문자열이지만 만료 시간, 탈취 대응, Secure 옵션 등 실제 서비스 수준의 세션 보안까지는 완벽하게 학습하지 못하였다.
- HTML을 문자열 치환으로 동적 렌더링하고 있어 구조가 복잡해질 경우 유지보수성이 떨어질 수 있다.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
- 로그인 상태를 클라이언트가 직접 들고 있는 값이 아니라, `sid` 쿠키와 서버의 세션 저장소를 연결해 판단하도록 했다.

- 그렇게 판단한 이유:
- HTTP는 stateless 프로토콜이기 때문에 요청마다 이전 로그인 상태를 자동으로 기억하지 않는다.
- 따라서 브라우저는 식별자 역할의 `sid`만 보내고, 실제 로그인 상태와 사용자 이름은 서버 메모리의 세션 저장소에서 확인하는 방식이 더 적절하다고 판단했다.

- 고려했지만 선택하지 않은 대안:
- 쿠키에 `username`이나 `logined=true`를 직접 저장하는 방식.
- 구현은 단순하지만 클라이언트가 값을 볼 수 있고 조작할 수 있어 인증 상태 판단 근거로 쓰기 부적절하다.
- 로그인 성공 후 정적 `index.html`을 그대로 내려주는 방식.
- 정적 파일만 사용하면 세션 상태에 따라 메뉴를 바꾸기 어렵기 때문에 `MainPageService`에서 서버 사이드로 HTML을 수정하는 방식을 선택했다.

- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
- 로그인 상태 판단 책임이 `Cookie` 값 자체가 아니라 `SessionStorage` 조회로 이동해 보안 모델이 더 명확해졌다.
- `MainPageServiceTest`를 통해 세션이 있을 때만 인사말과 로그인 사용자용 메뉴가 내려가는지 검증할 수 있게 됐다.
- 다만 HTML 문자열 치환 로직이 서비스 코드에 들어가면서, 화면 구조 변경 시 테스트와 치환 포인트를 같이 관리해야 하는 부담이 생겼다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
- 로그인 실패 후 query parameter를 붙여 `/login?error=...`로 이동했지만, 기대한 동적 응답이 나오지 않거나 테스트가 깨졌다.
- 또 세션 기반 메인 페이지를 추가한 뒤 `/index.html` 요청이 더 이상 `StaticService`가 아니라 `MainPageService`로 라우팅되어 기존 resolver 테스트가 실패했다.

- 처음 추측한 원인:
- query parameter 파싱이 안 되는 문제라고 생각했다.
- 또는 redirect 후 브라우저가 query string을 유지하지 못한다고 추측했다.

- 원인을 확인하기 위해 시도한 방법:
- `HttpRequestParser`에서 request target을 기준으로 path와 query를 분리하는 부분을 확인했다.
- `/login?error=invalid`, `/login/index.html?error=invalid` 같은 요청을 테스트에 넣어 path와 query parameter가 어떻게 분리되는지 검증했다.
- `HttpRequestResolver`가 query가 제거된 path만 보고 어떤 service로 라우팅하는지 확인했다.

- 실제 원인:
- query parameter 파싱 자체는 동작하고 있었다. 문제는 파싱 이후 라우팅이었다.
- 예를 들어 `/login/index.html?error=invalid`는 path가 `/login/index.html`로 분리되는데, service가 `/login`만 지원하면 해당 요청이 원하는 서비스로 가지 않는다.
- 이후 메인 페이지도 `/index.html`을 동적으로 처리하도록 바꾸면서 기존에는 static resource였던 path의 책임이 `MainPageService`로 이동했다.

- 해결 방법:
- 로그인 실패 redirect URL을 `/login?error=...`로 통일하고, `LoginPageService`가 `GET /login`을 처리하도록 정리했다.
- `MainPageService`를 resolver에 등록해 `/`와 `/index.html`을 명시적으로 처리하도록 했다.
- resolver 테스트의 기대값도 실제 라우팅 책임 변화에 맞춰 `StaticService`에서 `MainPageService`로 수정했다.

- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
- 먼저 HTTP request line의 request target이 parser에서 path/query로 어떻게 분리되는지 확인한다.
- 그 다음 분리된 path가 resolver에서 어떤 service로 매핑되는지 확인한다.
- 마지막으로 redirect 응답의 `Location` 값과 실제 GET 요청 path가 service와 명확하게 매핑되는 지 테스트 한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
- 기능을 구현한 뒤 매번 `./gradlew test`를 실행해 컴파일 오류와 기존 테스트 깨짐을 확인했다.
- 실패한 테스트를 단순히 기대값만 바꾸지 않고, 실제 책임이 이동한 코드 구조와 맞는지 확인했다.
- HTTP redirect, Cookie, Session, Routing을 각각 분리해서 원인을 추적했다.

- 그 접근이 도움이 된 이유:
- HTTP는 한 번의 함수 호출이 아니라 `요청 -> 응답 -> 브라우저의 다음 요청`으로 이어지는 흐름이라, 한 지점만 보면 원인을 잘못 판단하기 쉽다.
- parser, resolver, service, response를 단계별로 확인하니 “query parsing 문제”처럼 보였던 현상이 실제로는 routing 문제였음을 확인할 수 있었다.
- 테스트를 통해 `Content-Length`, query parameter, redirect `Location`, session cookie 같은 low-level HTTP 요소가 코드에서 어떻게 연결되는지 검증할 수 있었다.

- 다음에도 반복하고 싶은 점:
- HTTP 관련 문제는 먼저 raw request/response 관점에서 확인한다.
- redirect 문제는 status code와 `Location` 헤더를 먼저 본다.
- 로그인 상태 문제는 Cookie에 들어간 값과 서버 세션 저장소의 값을 함께 본다.
- service를 분리할 때는 resolver 테스트를 같이 수정해 라우팅 책임을 명확히 한다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
  - 로그인 실패 시 에러 메시지를 HTML에 동적으로 삽입하는 과정에서, HTML을 어떻게 다뤄야 하는지와 세션/쿠키 흐름을 한 번에 같이 고민하다 보니 문제를 작게 쪼개지 못했다.
  - 정적 HTML을 읽고, 쿼리 파라미터에 따라 alert 스크립트를 삽입하는 단순한 문제였는데, 이를 세션 관리와 동적 페이지 처리 구조까지 연결해 너무 큰 그림으로 접근했다.
  - 그 결과 먼저 직접 구현해보며 HTML 문자열 처리, 리소스 파일 읽기, 응답 생성 흐름을 차분히 확인하기보다 AI의 도움을 받아 빠르게 방향을 잡았다.

- 그렇게 된 원인:
  - 순수 자바로 웹서버를 구현하면서 Spring이나 템플릿 엔진이 자동으로 처리해주던 부분을 직접 구현해야 했고, HTML을 서버에서 동적으로 다루는 방식에 익숙하지 않았다.
  - 쿠키, 세션, 리다이렉트, HTML 파싱을 각각 분리해서 보지 못하고 한꺼번에 연결해서 생각하면서 구현 범위가 불필요하게 커졌다.

- 다음에는 다르게 해볼 구체적인 행동:
  - 비슷한 문제가 생기면 먼저 문제를 `파일 읽기 -> 문자열 변환 -> 특정 위치에 삽입 -> 응답 반환`처럼 작은 단계로 나누어 직접 구현해보겠다.
  - AI를 사용할 때도 바로 정답 코드를 받기보다, 내가 먼저 설계한 흐름과 막힌 지점을 설명하고 “어떤 개념을 확인해야 하는지”를 묻는 방식으로 학습 중심으로 활용하겠다.
