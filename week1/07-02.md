# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: MIME 타입을 학습하고, 정적 리소스 응답 시 파일 확장자에 따라 적절한 Content-Type을 내려주도록 구현했다.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 존재하지 않는 파일을 요청했을 때의 예외 처리와 404 응답 처리를 아직 마무리하지 못했다.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 추후 확장을 고려해 정적 리소스 응답 처리 구조에 템플릿 메서드 패턴을 적용해보았다.
- 그렇게 판단한 이유: HTML, CSS, JS, 이미지처럼 정적 리소스를 읽어 응답하는 흐름은 비슷하지만, 이후 JSON 응답이나 다른 형태의 리소스 응답이 필요해질 경우 일부 처리 방식만 달라질 수 있다고 판단했기 때문이다.
- 고려했지만 선택하지 않은 대안: 전략 패턴을 적용하는 것도 고려했다. 요청을 처리하는 큰 흐름은 같고, 일부 세부 처리만 달라지는 구조라서 템플릿 메서드와 전략 패턴 사이의 트레이드오프를 고민했다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 공통 처리 흐름을 한 곳에 모으고, 리소스 타입별로 달라지는 구현부만 재정의할 수 있게 되어 확장과 테스트가 쉬워졌다. 다만 타입마다 새로운 하위 객체를 만들어야 하므로 클래스 수가 늘어나고, 상속 구조에 의존하게 되는 점은 아쉬웠다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 정적 파일을 응답했을 때 브라우저가 리소스를 의도한 방식으로 해석하지 못할 수 있다는 점을 확인했다. HTML은 화면에 렌더링되어야 하고, CSS는 스타일로 적용되어야 하며, JS는 스크립트로 실행되어야 하는데, 이를 위해 응답 헤더의 MIME 타입이 중요하다는 것을 알게 되었다.
- 처음 추측한 원인: 브라우저가 파일 확장자만 보고 리소스 타입을 판단한다고 생각했다.
- 원인을 확인하기 위해 시도한 방법: HTTP 응답의 Content-Type헤더가 브라우저 동작에 어떤 영향을 주는지 학습하고, HTML, CSS, JS 파일을 각각 다른 MIME 타입으로 응답해야 한다는 점을 확인했다.
- 실제 원인: 브라우저는 단순히 파일 확장자만 보는 것이 아니라 서버가 응답한 Content-Type을 기준으로 리소스를 해석한다. 따라서 서버가 MIME 타입을 제대로 내려주지 않으면 브라우저가 해당 파일을 HTML, CSS, JS, 이미지 등으로 올바르게 처리하지 못할 수 있다.
- 해결 방법: 요청된 파일의 확장자를 기준으로 text/html, text/css, application/javascript, image/png 등의 MIME 타입을 매핑하고, 응답 헤더에 Content-Type을 포함하도록 구현했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 브라우저에서 리소스가 제대로 적용되지 않을 때는 파일 내용뿐 아니라 HTTP 응답 헤더, 특히 Content-Type, 상태 코드, 경로, 실제 응답 바디를 먼저 확인해야겠다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 기능을 구현하기 전에 브라우저가 MIME 타입을 어떻게 해석하는지 먼저 학습하고, 이를 코드 구조에 반영하려고 했다.
- 그 접근이 도움이 된 이유: 단순히 “파일을 읽어서 응답한다”는 수준에서 끝내지 않고, 실제 브라우저가 서버 응답을 어떻게 사용하는지 이해할 수 있었다. 덕분에 HTTP 응답 헤더의 역할을 더 구체적으로 이해하게 되었다.
- 다음에도 반복하고 싶은 점: 구현 중 막히는 부분이 생기면 바로 코드를 고치기보다, 해당 기능이 동작하는 원리를 먼저 확인하고 그 근거를 바탕으로 구조를 잡는 습관을 유지하고 싶다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 정상 동작을 확인하는 데 매몰되어, 실패할 수 있는 요청을 먼저 떠올리지 못했다. 특히 파일을 읽는 기능은 존재하지 않는 경로, 지원하지 않는 확장자, 읽기 실패 같은 예외가 자연스럽게 발생할 수 있는데도, 정상 파일 응답 흐름만 기준으로 코드를 작성했다.
- 그렇게 된 원인: 기능을 “일단 동작하게 만들기”에 매몰되면서, 요청이 실패했을 때 서버가 어떤 응답을 내려야 하는지까지 함께 설계하지 못했다. 또한 MIME 타입 처리와 템플릿 메서드 구조 적용에 집중하다 보니, 예외 상황을 테스트 케이스로 먼저 분리해 확인하는 행동이 부족했다.
- 다음에는 다르게 해볼 구체적인 행동: 파일 I/O나 HTTP 응답처럼 실패 가능성이 큰 기능을 구현할 때는 정상 케이스를 구현하기 전에 실패 케이스를 먼저 적어두겠다.
