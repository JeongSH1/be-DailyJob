# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: HTTP 구조 학습, HTTP을 담는 객체 구현, raw HTTP 파서 구현
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 현재는 index.html만 반환할 수 있게 임시로 구현했지만 다양한 타입을 반환할 수 있도록 확장성을 고려한 코드를 구현하지 못했다. 다양한 타입을 지원하면서, 요구사항에 따른 코드 변경이 최소화 될 수 있는 구조를 생각하고 있다.

###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: raw HTTP에 대한 텍스트를 미리 예시로 가져와 파싱이 잘 이루어지는지 간단한 단위 테스트를 작성했다.
- 그렇게 판단한 이유: 코드가 변경될 때 마다, 직접 들어오는 입력을 확인하기 위해 콘솔에 입력하고 검증하는 과정이 번거롭다고 판단하였다.
- 고려했지만 선택하지 않은 대안: 메서드, 자원 위치를 입력받아, 동적으로 raw HTTP을 생성하는 테스트 기능까지 구현할까 하였지만 금일 목표를 달성하는데에는 현재 상태로 충분하여 추후 할 일로 분류하였다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: HTTP 파싱 로직이 변경될 때마다, 이번에 작성한 테스트 코드를 실행하면 코드가 올바르게 동작하는지 빠르게 확인할 수 있다.

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 기존에 "./static" + 요청 경로 방식으로 매핑했지만 정적 리소스를 읽어 오지 못했다.
- 처음 추측한 원인: 정적 파일을 찾는 기준 경로를 잘못 설정했거나, 빌드 결과물 기준의 상대 경로를 잘못 이해했다고 생각했다.
- 원인을 확인하기 위해 시도한 방법: src/main/resources/static을 기본 경로에 포함하여 파일을 읽어오도록 하였다.
- 실제 원인: 실행 시점의 현재 작업 디렉터리와 소스 코드상의 리소스 경로가 달라 "./static" 경로에서 파일을 찾을 수 없었다.
- 해결 방법: 개발 과정에서는 "src/main/resources/static" 를 포함한 경로를 사용하면 잘 동작하지만 빌드된 jar 파일로 실행하면 파일을 찾지 못할 것이다. 개발/배포 환경에서 변경되는 정적 리소스 경로를 일관되게 읽어오게 하고 싶었다. 그리하여 AI에게 해당 문제를 해결하는 법을 물어보았고, 런타임에 classPath 경로를 통해 일관되게 파일을 읽어오는 법을 찾아 구현하였다. 이 과정에서 ClassLoader 과정에 대해 학습할 수 있었다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 파일 경로처럼 개발 환경과 배포 환경의 차이로 인해 문제가 발생할 수 있는 부분은, 실행 환경에 의존하지 않도록 일관된 기준을 정하거나 환경별로 적절히 분기할 수 있는 방법을 먼저 확인해야 할 듯 하다.

4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
    HTTP 요청 구조를 먼저 이해한 뒤, 요청 라인과 헤더를 직접 파싱해 보며 서버가 요청을 해석하는 흐름을 구현해 보았다.
- 그 접근이 도움이 된 이유:
    단순히 프레임워크가 제공하는 기능을 사용하는 것이 아니라, HTTP 요청이 문자열 형태로 들어오고 이를 서버가 어떻게 해석하는지 직접 확인할 수 있었다.
- 다음에도 반복하고 싶은 점:
    새로운 개념을 학습할 때 바로 추상화된 도구를 사용하기보다, 먼저 기본 구조와 동작 원리를 작게 구현해 보면서 이해하는 방식을 반복하고 싶다.

5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
    HTTP 파싱 로직을 구현하면서 복잡한 문자열 처리나 예외 케이스를 충분히 이해하기 전에 AI가 제공한 코드를 따라 구현한 점이 아쉬웠다.
- 그렇게 된 원인:
    정규식이나 문자열 분리처럼 세부 구현이 번거롭게 느껴졌고, 이 부분이 현재 학습 목표와 직접적으로 관련이 적다고 판단해 빠르게 AI의 도움을 받았다.
- 다음에는 다르게 해볼 구체적인 행동:
    AI가 작성한 코드를 그대로 사용하기 전에, 먼저 내가 처리해야 할 입력과 출력, 예외 상황을 정리한 뒤 AI에게 구현을 요청하고, 결과 코드를 한 줄씩 설명할 수 있는지 확인하겠다.

###### 6. 전체 회고
오늘 AI를 쓰지 않고 자력으로 학습해서 구현하려고 했는데, HTTP 파싱 로직을 구현하면서 복잡하고 까다로운 문자열 처리에 대해 AI가 제공해준 코드를 따라 구현했다.
