# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 자체 DB는 내가 처음부터 설계해 구현하기에는 난도가 너무 높아 AI의 설명과 코드 제안을 중심으로 학습하고 구현했다. 그 과정에서 JDBC 인터페이스를 직접 구현하며 DriverManager가 URL에 맞는 Driver를 찾고, Driver가 Connection을 생성해 애플리케이션과 DB 엔진을 연결하는 구조를 학습했다. 또한 커넥션 풀, 쿼리 파싱과 실행, 저장소의 책임과 인메모리 데이터가 CSV에 저장되는 흐름을 확인했다.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: AI가 제안한 구조와 코드를 따라가며 기능은 완성했지만, 전체 동작을 도움 없이 다시 설계하거나 각 구현의 필요성을 모두 설명할 수준까지 이해하지는 못했다. 특히 커넥션 풀, 트랜잭션, 저장소가 실제 DB에서 어떻게 맞물리는지는 더 학습해야 한다.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: DB 전체를 자력으로 구현하는 데 집착하기보다 AI를 적극적으로 사용해 먼저 동작하는 구조를 만들고, 변경 과정을 따라가며 각 계층의 책임을 학습하기로 했다. 구조적으로는 애플리케이션이 DB 내부 객체를 직접 알지 않고 DriverManager, Driver, Connection으로 이어지는 표준 JDBC 계약과 하나의 DataSource를 통해 접근하도록 정리했다.
- 그렇게 판단한 이유: SQL 파서, JDBC 인터페이스, 커넥션 풀, 트랜잭션, 파일 저장을 동시에 직접 구현하기에는 현재 이해도와 시간 모두 부족했다. 막연히 오래 붙잡기보다 AI가 제안한 구현을 테스트하고 수정 이유를 질문하되, Real MySQL 8.0도 함께 확인하며 실제 DB의 구조와 비교하는 편이 학습에 도움이 된다고 판단했다.
- 고려했지만 선택하지 않은 대안: 기존 H2 DAO 구조를 그대로 자체 DB에 맞추거나 문서 API와 SQL API를 함께 유지하는 방법도 있었지만, 학습 대상과 사용 경로가 겹치고 책임이 모호해져 사용하지 않는 문서 API를 제거했다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: AI의 도움으로 짧은 시간에 넓은 범위를 구현하고 테스트할 수 있었지만, 코드의 규모가 내 이해 속도보다 빠르게 커졌다. 구조를 여러 번 정리하며 계층의 경계는 확인했지만, 이후에는 각 계층을 작은 단위로 다시 구현해 보며 이해하지 못한 부분을 검증해야 한다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: AI가 빠르게 만들어 준 코드의 양이 많아지자 기능은 동작하지만 내가 전체 흐름과 각 객체가 필요한 이유를 설명하기 어려워졌다.
- 처음 추측한 원인: DB 자체가 어려운 주제이므로 일단 완성된 코드를 여러 번 읽으면 자연스럽게 이해할 수 있을 것이라고 생각했다.
- 원인을 확인하기 위해 시도한 방법: AI에게 패키지와 객체의 책임을 다시 나누도록 요청하고, DriverManager가 드라이버를 발견하는 과정부터 Driver가 Connection을 생성하고 쿼리가 엔진으로 전달되는 흐름을 테스트와 함께 따라가 보았다. AI의 설명만 받아들이지 않기 위해 실제 MySQL이 요청을 처리하고 데이터를 관리하는 구조도 함께 확인했다.
- 실제 원인: DB 개념이 어려운 것뿐 아니라 한 번에 너무 많은 구현을 AI에게 맡겨, 내가 먼저 입력과 출력 및 책임을 예상해 보는 과정이 부족했다.
- 해결 방법: Real MySQL 8.0에서 확인한 실제 DB의 역할 구분을 참고해 현재 코드의 흐름을 다시 바라봤다. JDBC 계층에서는 드라이버 등록과 URL 판별, Connection 생성의 책임을 구분하고, Connection 구현이 트랜잭션 상태와 쿼리 실행의 진입점을 관리하도록 정리했다. 엔진에서는 파서가 쿼리 모델을 만들고 실행기가 저장소를 사용하도록 분리했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 클래스 이름이나 패키지를 먼저 바꾸기보다 시스템 외부에서 들어오는 지점, 각 객체의 입력과 출력, 서로 다른 변경 이유를 먼저 한 장에 적어보겠다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 혼자 해결한 것처럼 넘기지 않고 이해하기 어려운 부분은 AI에게 계속 질문했으며, 제안받은 구조를 실제 DB의 동작과 비교하려고 했다.
- 그 접근이 도움이 된 이유: 이전에는 JDBC를 DAO에서 SQL을 실행하기 위한 API 정도로만 생각했지만, Driver와 Connection을 직접 구현하면서 JDBC가 애플리케이션과 서로 다른 DB 구현을 분리하는 계약이라는 점을 이해했다. 코드와 책을 함께 보며 연결 생성, 트랜잭션 관리, 쿼리 실행이 왜 서로 다른 책임인지도 확인할 수 있었다.
- 다음에도 반복하고 싶은 점: 난도가 높은 주제에서는 AI의 도움을 적극적으로 받되, 한 단계가 끝날 때마다 내가 이해한 흐름을 먼저 설명하고 테스트가 무엇을 보장하는지 확인하겠다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: DB 구현이 어렵다는 이유로 AI에게 한 번에 큰 범위의 구현과 구조 개선을 요청했다. 그 결과 같은 날 패키지와 공개 구조를 여러 차례 바꾸었고, 일부 코드는 내가 충분히 이해하기 전에 테스트 통과만 확인하고 넘어갔다.
- 그렇게 된 원인: 직접 구현하면 목표를 끝내지 못할 것이라는 부담이 컸고, AI가 빠르게 작성하는 코드를 보며 학습 속도도 그만큼 빨라질 것이라고 생각했다. 하지만 코드를 얻는 속도와 내가 이해하는 속도는 같지 않았다.
- 다음에는 다르게 해볼 구체적인 행동: AI에게 완성 구현을 요청하기 전에 내가 이해한 요구사항과 책임을 먼저 적겠다. 제안받은 코드는 파서, 실행기, 저장소처럼 한 단위씩 설명하고 일부를 직접 다시 작성한 뒤 다음 단계로 넘어가겠다.

### 6. 전체 회고
오늘은 자체 DB라는 어려운 주제를 AI 위주로 학습하고 구현했다. 혼자 처음부터 만들었다면 오늘 안에 이 범위까지 도달하기 어려웠을 것이고, AI 덕분에 DriverManager부터 Driver, Connection, 쿼리 실행과 파일 저장까지 이어지는 흐름을 빠르게 경험할 수 있었다. 여기에 실제 MySQL의 DB 내부 구조도 함께 확인하면서 현재 만든 단순한 구조와 비교해 보았다.

다만 AI가 구현한 코드가 빠르게 늘어나는 동안 내 이해가 따라가지 못했고, 테스트가 통과한다는 사실을 이해의 증거처럼 받아들인 점은 아쉽다. 책과 JDBC 구현을 함께 확인한 뒤에는 DB 구조를 이해하려면 클래스만 읽는 것이 아니라 드라이버 발견, Connection 생성, 트랜잭션, 쿼리 처리, 저장 방식이 어떤 책임으로 연결되는지 함께 봐야 한다는 것을 알게 됐다.

앞으로도 어려운 구현에는 AI를 사용하되 완성 속도를 학습 속도로 착각하지 않겠다. AI에게 구조를 질문한 뒤에는 실제 공식 문서의 자료에서 근거를 다시 확인하고, DriverManager부터 Driver, Connection, DB 엔진으로 이어지는 흐름을 내 말로 정리하겠다. 오늘 만든 DB도 결과물로 끝내지 않고 각 계층이 왜 필요한지 설명할 수 있을 때까지 다시 확인하겠다.
