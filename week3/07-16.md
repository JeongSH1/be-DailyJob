# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 업로드 요청 경로가 허용된 디렉터리 밖으로 벗어날 수 없도록 검증하고, 이미지가 없는 게시글도 저장될 수 있게 요구사항을 다시 맞췄다. 또한 애플리케이션을 다시 초기화할 때 기존 데이터를 목데이터로 덮어쓰던 문제를 수정하며 경로 보안과 초기화의 멱등성을 학습했다.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 확인된 사례는 수정했지만 파일 경로와 데이터 초기화에서 어떤 입력과 실행 시점을 신뢰하면 안 되는지 체계적으로 정리하지 못했다. 정상 동작뿐 아니라 심볼릭 링크, 일부 테이블만 비어 있는 상태처럼 더 복잡한 경계 조건도 추가로 검토해야 한다.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 초기화 코드를 실행할 때마다 목데이터를 넣지 않고, 핵심 테이블이 모두 비어 있을 때만 최초 데이터를 삽입하도록 조건을 두었다.
- 그렇게 판단한 이유: 스키마 준비와 목데이터 삽입을 같은 초기화 과정으로 보면서 재실행의 영향을 생각하지 못했고, DAO가 다시 생성될 때 사용자가 수정한 프로필 이미지가 초기값으로 돌아갔다. 초기화는 여러 번 호출되어도 이미 저장된 데이터를 변경하지 않아야 한다고 판단했다.
- 고려했지만 선택하지 않은 대안: 목데이터 삽입 중 발생하는 중복 오류를 무시하거나 항상 기존 행을 갱신하는 방법도 있지만, 오류로 흐름을 제어하거나 실제 데이터를 초기값으로 덮어쓸 위험이 있어 선택하지 않았다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 초기화의 실행 조건이 코드로 드러났고 재생성 이후에도 변경 데이터가 유지되는지를 회귀 테스트로 남겼다. 다만 모든 핵심 테이블이 비어 있어야 한다는 현재 기준이 부분 초기화 상황에도 적합한지는 더 고민해야 한다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 업로드 경로에 `..`이 포함되면 전용 업로드 디렉터리 밖의 파일을 읽을 수 있었고, DAO를 다시 생성하면 사용자가 변경한 데이터가 사라졌다.
- 처음 추측한 원인: 경로를 `normalize`하면 상위 경로 접근이 제거될 것이고, 초기화 SQL은 필요한 데이터가 이미 있으면 자연스럽게 유지될 것이라고 막연히 생각했다.
- 원인을 확인하기 위해 시도한 방법: 업로드 디렉터리 밖에 비밀 파일을 만든 뒤 경로 탈출 요청으로 읽히는지 테스트했고, 사용자 프로필 이미지를 변경한 후 DAO를 다시 만들어 값이 유지되는지도 테스트했다.
- 실제 원인: 경로를 정규화하는 것만으로는 최종 경로가 허용된 디렉터리 안에 있는지 보장할 수 없었다. 데이터 초기화도 실행될 때마다 목데이터 SQL을 적용해 기존 값을 덮어쓰고 있었다.
- 해결 방법: 기준 업로드 경로에 요청 경로를 결합하고 정규화한 뒤 결과가 기준 경로로 시작하는지 검증했다. 데이터베이스는 핵심 테이블의 행 수를 조회해 모두 비어 있을 때만 목데이터를 넣도록 변경했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 입력값을 변환하거나 정규화했다는 사실만으로 안전하다고 생각하지 않고, 변환된 최종 결과가 허용 범위 안에 있는지 검증하겠다. 초기화 로직은 두 번 실행했을 때 기존 상태가 보존되는지도 먼저 테스트하겠다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 눈앞의 조건문만 고치지 않고 실제로 보호해야 하는 파일과 유지되어야 하는 데이터를 준비해 회귀 테스트를 작성했다.
- 그 접근이 도움이 된 이유: 단순히 404가 반환되는지나 초기화가 성공하는지만 확인하지 않고, 외부 파일의 내용이 노출되지 않는지와 사용자가 변경한 값이 재생성 후에도 남는지를 동작 기준으로 검증할 수 있었다.
- 다음에도 반복하고 싶은 점: 보안과 데이터 보존 문제는 구현 세부사항보다 공격하거나 잃어버릴 실제 대상을 테스트에 먼저 표현하겠다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 정상적인 파일명과 최초 실행만 기준으로 구현해 외부 입력이 경로를 조작할 가능성과 초기화가 반복 호출될 가능성을 미리 떠올리지 못했다. 이미지가 반드시 존재한다고 가정해 선택 입력을 필수 조건처럼 처리한 점도 요구사항 확인이 부족했다.
- 그렇게 된 원인: 기능이 한 번 성공하는지에 집중했고, 같은 코드가 악의적인 입력이나 두 번째 실행을 만났을 때 어떤 상태를 만드는지 생각하지 않았다. 구현 전에 신뢰할 수 없는 입력과 보존해야 할 상태를 구분하지 않은 것이 원인이었다.
- 다음에는 다르게 해볼 구체적인 행동: 파일 시스템 코드는 기준 디렉터리 밖으로 나가는 입력을 먼저 만들고, 초기화 코드는 최소 두 번 실행하는 테스트부터 작성하겠다. 선택 입력은 비어 있음, 이름만 있음, 데이터만 없음의 경우를 요구사항과 함께 확인하겠다.

### 6. 전체 회고
오늘은 기능이 정상적으로 한 번 동작하는 것과 안전하게 반복 동작하는 것은 다른 문제라는 점을 배웠다. `normalize`를 호출했다는 이유로 경로가 안전하다고 생각했고, 초기화라는 이름 때문에 여러 번 실행되어도 기존 데이터가 유지될 것이라고 확인 없이 가정했다.

두 문제 모두 API 사용법보다 검증 기준을 잘못 세운 것이 핵심이었다. 경로는 문자열을 정리했는지가 아니라 최종 위치가 허용된 디렉터리 안인지 확인해야 했고, 초기화는 SQL이 성공했는지가 아니라 이미 존재하는 상태를 훼손하지 않는지 확인해야 했다.

앞으로는 외부 입력을 다룰 때 허용 범위를 먼저 정의하고, 상태를 만드는 코드는 재실행 결과까지 설계하겠다. 테스트도 정상 입력만 재현하지 않고 공격 가능한 입력과 기존 데이터가 있는 상황을 먼저 만들어 내 가정이 실제로 안전한지 검증하겠다.
