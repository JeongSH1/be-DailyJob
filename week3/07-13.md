# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - 좋아요 클릭 수를 일정 시간 동안 누적해 한 번의 요청으로 전송하도록 클라이언트 디바운싱을 구현했다.
  - 좋아요 증가 쿼리를 `likes = likes + ?` 형태로 변경해 애플리케이션에서 기존 값을 계산하지 않고 DB가 원자적으로 증가시키도록 정리했다.
  - 좋아요 변경 후 DB와 캐시 값이 달라지던 문제를 수정하고, Cache-Aside와 Write-Through의 차이, 캐시 갱신과 무효화 전략을 학습했다.

- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - DB 변경 성공 후 캐시 갱신이 실패했을 때의 복구 정책을 명확히 정하지 못했다.
  - 현재 오토커밋 환경에서 여러 SQL을 하나의 트랜잭션과 같은 커넥션으로 묶는 구조는 구현하지 못했다.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 좋아요 수를 먼저 조회해 애플리케이션에서 계산한 뒤 저장하지 않고, `UPDATE article SET likes = likes + ?` 한 문장으로 증가시키기로 했다.

- 그렇게 판단한 이유: 조회와 수정이 분리되면 두 요청이 같은 기존 값을 읽은 뒤 같은 결과를 저장해 증가분이 유실될 수 있다. 현재는 트랜잭션을 적용하지 않은 오토커밋 환경이므로, `SELECT ... FOR UPDATE`로 조회하더라도 조회 문장이 끝나는 순간 락이 해제되어 다음 `UPDATE`까지 보호하지 못한다. 반면 `likes = likes + ?`는 하나의 SQL 안에서 행 락과 계산, 변경이 이루어져 동시 요청에서도 증가분을 안전하게 반영할 수 있다.

- 고려했지만 선택하지 않은 대안: `SELECT ... FOR UPDATE`로 현재 좋아요 수를 조회한 뒤 `UPDATE`하는 방법을 고려했다. 그러나 이 방식은 두 쿼리가 반드시 하나의 트랜잭션과 같은 커넥션에서 실행되어야 의미가 있으며, 현재 구조에서는 이를 보장하지 못해 선택하지 않았다. SQL 문자열에 `BEGIN`, `UPDATE`, `SELECT`, `COMMIT`을 함께 넣는 방법도 생각했지만 JDBC의 다중 쿼리 처리와 결과 반환이 불명확하고 트랜잭션 책임도 흐려져 사용하지 않았다.

- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 조회와 계산 로직이 사라져 코드가 단순해졌고, 동시 요청에 대한 안전성을 애플리케이션 코드가 아니라 DB의 원자적 연산에 맡길 수 있게 됐다. 이후 테스트에서도 여러 요청이 동시에 증가했을 때 최종 합계가 유실되지 않는지를 명확한 기준으로 확인할 수 있게 됐다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 좋아요를 누른 직후에는 증가한 값이 보였지만, 페이지를 새로고침하면 이전 좋아요 수가 다시 표시됐다.

- 처음 추측한 원인: 클라이언트 디바운싱 과정에서 누적한 `count`가 누락됐거나, 좋아요 증가 쿼리가 정상적으로 실행되지 않은 것으로 생각했다.

- 원인을 확인하기 위해 시도한 방법: 클라이언트가 전송한 `count`, DB에 저장된 좋아요 수, 새로고침 후 화면에 표시된 값을 각각 비교했다. DB 값은 정상적으로 증가한 것을 확인한 뒤, 메인 페이지가 좋아요 수를 DB에서 직접 읽는지 캐시에서 읽는지 조회 경로를 따라가며 확인했다.

- 실제 원인: 좋아요 변경 서비스는 DB만 갱신했지만, 메인 페이지 조회 서비스는 이전 값이 남아 있는 캐시를 사용하고 있었다. 읽기에서는 캐시를 먼저 확인하고 없으면 DB를 조회하는 Cache-Aside 방식과 유사하게 동작했지만, 쓰기 시 캐시를 갱신하거나 무효화하지 않아 불일치가 발생했다.

- 해결 방법: 좋아요 수를 DB에서 원자적으로 증가시킨 후 캐시에도 최신 값을 반영하도록 수정했다. 이 과정에서 현재 구현은 서비스가 DB와 캐시를 각각 직접 수정하는 방식이므로 완전한 Write-Through는 아니라는 점을 확인했다. 또한 캐시를 직접 갱신하면 즉시 최신 값을 제공할 수 있지만, DB 변경 성공 후 캐시 갱신이 실패할 수 있어 캐시를 삭제하고 다음 조회에서 다시 적재하는 방식도 대안으로 검토했다.

- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 먼저 데이터의 최종 원본이 어디인지, 읽기와 쓰기 경로가 각각 DB와 캐시를 어떤 순서로 거치는지 그려보겠다. 그다음 DB와 캐시를 하나의 트랜잭션으로 묶을 수 없는 상황에서 갱신과 무효화 중 어떤 전략이 실패에 더 안전한지 확인하겠다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 화면에 잘못 표시되는 값만 수정하지 않고, 클라이언트 디바운싱부터 DB 증가 쿼리, 오토커밋과 행 락, 캐시 조회까지 전체 변경 흐름을 따라가며 문제를 확인했다.

- 그 접근이 도움이 된 이유: 좋아요 문제를 단순한 캐시 갱신 누락으로만 보지 않고, 동시성 문제와 트랜잭션 경계까지 함께 이해할 수 있었다. `SELECT ... FOR UPDATE`는 쿼리 자체가 아니라 트랜잭션 종료 시점까지 락을 유지하는 것이 핵심이며, 오토커밋 환경에서는 조회 후 수정 흐름을 보호할 수 없다는 점도 확인했다.

- 다음에도 반복하고 싶은 점: 기능을 구현할 때 SQL 문장 하나만 보는 것이 아니라 커넥션, 오토커밋, 락 유지 시점, 캐시 갱신 순서까지 함께 보겠다. 새로운 기술 용어를 학습하는 데 그치지 않고 현재 코드가 실제로 Cache-Aside인지, Write-Through인지, 또는 두 전략이 섞인 구조인지 직접 대입해 설명하겠다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 처음에는 캐시를 조회 속도를 높이는 저장 공간으로만 생각했고, DB를 변경한 뒤 캐시를 어떻게 처리할지까지 설계하지 않았다. 또한 `SELECT ... FOR UPDATE`를 사용하면 자동으로 조회와 수정 사이가 보호될 것이라 생각해, 오토커밋과 동일 커넥션 조건을 뒤늦게 확인했다.

- 그렇게 된 원인: SQL 문법과 개별 기능에 집중하면서 트랜잭션이 여러 SQL을 하나의 작업 단위로 묶고 같은 커넥션을 유지한다는 전제를 충분히 고려하지 못했다. 캐시도 읽기 전략만 생각하고 쓰기 실패나 DB와 캐시 사이의 부분 실패 가능성을 함께 보지 못했다.

- 다음에는 다르게 해볼 구체적인 행동: DB 변경 로직을 작성하기 전에 해당 작업이 SQL 한 문장으로 원자적으로 처리 가능한지 먼저 확인하겠다. 여러 SQL이 필요하다면 오토커밋 여부, 같은 커넥션 사용, 커밋과 롤백 지점을 먼저 설계하겠다. 캐시는 Cache-Aside, Write-Through, Write-Behind 중 어떤 전략을 적용하는지 명시하고, DB 성공·캐시 실패 상황에서는 캐시 갱신보다 무효화를 우선할지 기준을 세워 테스트하겠다.
